    This module is the main module of monoco.
    It inits monoco metamodel and loads monoco core system.

    This module provides the following classes:

    This module is a rollup of the following modules:

    • monoco-behavior
      This module manages the behaviors of all components. A behavior is a mecanism that allow users to add actions that will be executed when a specific state of a component will change.
    • monoco-channel
      This module manages the channel.
    • monoco-component
      This module manages the components. It is the factory of all the components that are created by monoco.
    • monoco-db
      This module manages monoco database.
      monoco database is a micro NoSQL Database that contains:
      - collections to store documents (schemas, types, components, ...) and
      - APIs to import or export documents.
      monoco Database is closely linked to monoco metamodel and monoco components because:
      - all operations done by monoco database must be compliant with the model before being finished,
      - insert operation automatically creates a component and
      - remove operation automatically destroy a component.
    • monoco-helper
      This module contains all the functions used by all the modules.
    • monoco-log
      This module contains all the functions that write a log.
    • monoco-metamodel
      This module manages monoco metamodel.
      monoco metamodel loads schemas and types, analyzes them and creates the component classes and related MonocoDatabaseCollections.
    • monoco-state
      This module manages the states of all thecomponents.
    • monoco-system
      This module contains monoco core system.
    • monoco-workflow
      This module manages the workflow of monoco. It behaves like a workflow engine.
      It checks if the change of status of a component is valid to be executed. By valid, it means that:
      - the state is valid for the component,
      - the input (i.e. parameters) of all actions for the state are compliants with the model and
      - the output of all actions are compliants with the model.
      If an error occurs, the workflow will call the error state of the component and monoco.
      If the error can break the consistency of the current system, the worklow will stop.
